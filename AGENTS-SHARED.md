# AGENTS-SHARED.md — 全プロジェクト共通ルール(単一の真実源)

このファイルは **このリポジトリの全プロジェクトに共通する、プロジェクト非依存のルール**だけを置く。
各プロジェクトの `CLAUDE.md` は冒頭の「参照宣言」でこのファイルを `read-only` 宣言し、`@` import で取り込む。

> **このファイルの編集は特別扱い。** ここを変えると全プロジェクトに波及する。
> 編集する前に必ず「**これは本当に横断ルールか? 今いるディレクトリ固有の話ではないか?**」を確認し、
> 横断であることが確実で、かつユーザーの承認を得たときだけ編集する。既定は **read-only**。
>
> **書き込み前の必須判断(レビュー等とは無関係の前提条件)**: 書こうとしている事柄が
> **恒久的な共通ルール**なのか、**今回限り(その場・現作業限定)の判断**なのかを、**必ずエージェント自身が判断する**。
> 今回限りの判断は**共通ルールに書かない**(その場の指示・作業メモ・PR 説明に留める)。恒久ルールだけをここに書く。
> 個別プロジェクトの話・具体値・例(タグ名/ボード名/ピン/PREFIX/画面構成 など)は**絶対に書かない**
> ——書いた瞬間に別プロジェクトへその情報が漏れる。具体値は各プロジェクトの `CLAUDE.md` に書くこと。
>
> 判定の一言ルール: **「別プロジェクトにそのまま貼れる内容か?」** 貼れる=ここ。貼れない=各 `CLAUDE.md`。

## 1. 作業境界(全プロジェクト共通の前提)

- AI エージェントは、**最初に開いた作業ディレクトリ「以下」しか見ない**。親や兄弟ディレクトリを勝手にたどらない。
- 例外は、各プロジェクト `CLAUDE.md` の「参照宣言 (refs)」に**明示列挙されたパスだけ**。宣言された権限
  (`read` / `read-only`)の範囲で参照してよい。列挙が無いものは読まない(ノイズと情報漏れを防ぐため)。
- `read-only` 宣言された参照先は、**指示があっても書き換えない**(参照専用)。

## 2. リポジトリ全体の不可侵ルール

- **このリポジトリの可視性(public / private)は、リポジトリ直下の `CLAUDE.md` に明記された方針に従う。**
  方針が書かれていない場合は勝手に変更せず、ユーザーに確認する。private 指定なら**勝手に public 化しない**。
- コミット履歴は人間の作業として保つ。**AI 生成の署名・`Co-Authored-By` 行は付けない**(必須)。

## 3. コミット / ブランチの慣習

### コミット
- **コミット本文は日本語**。Conventional Commits 風の接頭辞(`feat:` `fix:` `refactor:` `chore:` `release:` 等。
  ドキュメント改修は `chore:` に寄せる)。
- **作業中はこまめに commit & push する**。いつデータが消えても復旧できるよう、まとめて1回ではなく
  意味のある最小単位ごとに push まで行う。
- ビルドが通っただけで満足しない。可能な限り実機(or 起動・操作・スクショ)で確認してからコミットする。

### ブランチ命名規則
形式は **`<type>/<説明>`**(必要に応じて scope を挟む。下記)。
- 使える文字は **小文字英数字 + ハイフン**。ドットは**バージョン番号のみ**(`release/...-v2.0.0`)。
- 禁止: 大文字・日本語・アンダースコア・スペース・連続ハイフン・先頭/末尾ハイフン。50〜60字以内が目安。
- 接頭辞(type): `feature` / `fix` / `hotfix` / `refactor`(動作を変えない改善) / `chore`(ドキュメント改修・雑作業・設定・版上げ) /
  `release` / `test` / `ci` / `perf` / `issue`(課題対応) / `future`(将来・統合) / `project`(プロジェクト/取り組み単位)。
- `issue` / `future` / `project` はいずれも同粒度の接頭辞。**先頭に番号を付けない**(`1-...` のような形は禁止)。
- 複数ブランチに分かれる作業では **scope = プロジェクト名(slug)** を挟む: `<type>/<project-slug>-<説明>`。
  対象プロダクト名は上位の `project/<slug>` ブランチ側で表現し、配下の作業ブランチは同じ slug を scope として共有する。

### ブランチ運用フロー
1. `main` で調査・要件定義(設計・仕様固め)を行う。
2. 作業が**何本のブランチに分かれるか**を見積もり、ユーザーに提示する。
3. ユーザーの OK を得てから作成に進む。
4. 分岐:
   - **作業ブランチが 2 本以上** → `main` から `project/<slug>` を作成し、その配下に `feature`/`fix` 等を切る
     (`project/<slug>` から分岐。各作業ブランチは `<type>/<slug>-<説明>`)。
   - **1 本で済む軽微な変更**(文言修正・ドキュメント更新など) → `project` は作らず、`main` から直接 1 本
     (`chore`/`fix` 等)を切る。

### AI モデル名の表記(スナップショット — 陳腐化する前提で更新する)
ブランチ名・エージェント名・プロジェクト名に AI モデルを含めるときは、**正規トークン(小文字 ASCII)**を使う。
音声入力等でカタカナ化・誤記したら正規トークンへ直す(直す案を提示する)。
- このリストは**時点スナップショット**。新モデルが出たら追記し、廃止モデルは「廃止」表記で残す。
- 正規化の方針例: カタカナ表記・スペース・大文字混じりは、対応する小文字 ASCII の正規トークンへ寄せる。

## 4. リリースと版上げ(原則 — 仕組みは共通、具体値は各プロジェクト)

- リリースは「版が上がったときだけ」成果物をビルド・公開する**版ゲート**で守る(採用する場合)。
- **したがって、ソースを直しても版を上げなければビルドも公開もされない。** 挙動が同じリファクタでも、
  出荷ファイル(成果物)を作り直すなら**版を上げる**。
- 採番の目安: PATCH=軽微 / MINOR=機能追加・バグ修正・出荷ファイルが変わる変更(通常はここ) / MAJOR=互換に関わる大変更。
- 各成果物の版の出どころ・タグ接頭辞・対応プラットフォームは**各プロジェクトの `CLAUDE.md`** と
  リポジトリ直下の対応表に書く(このファイルには具体値を書かない)。

## 5. リリースの完了定義(「リリースしました」で終わらせない)

成果物をリリースしたら、次の手順を**最後まで**行ってから完了とする:

1. CI(GitHub Actions)の run が**成功**し、対象タグの Release が**作成**されたことを確認する。
2. Release の**アセット数を検証**する(期待数は各プロジェクトの `CLAUDE.md` 参照)。不足していたら原因を調べる。
3. 成果物を**実際にダウンロード**して、取得できることを確認する。
4. **ユーザーに開いた状態で提示**し、最終確認を促す(GUI なら起動して見せる、ファームなら成果物を提示)。

検証が通らなかったら「完了」と言わない。失敗はその出力とともに正直に報告する。

## 6. ホストテスト先行の原則

- ライブラリやプロファイル(出力バイト列に影響するもの)を変えたら、まず環境非依存の**ホストテストで一致を確認**してから実機へ進む。
- ホストテストの具体的な場所(`test/host` 等)は各プロジェクトの `CLAUDE.md` に書く。

## 7. 環境の共通の罠(必要なら各プロジェクトで追記)

- 開発環境固有の罠(エンコーディング・シェルの挙動・ツールチェーンのクセ 等)は、横断で常に当てはまるものだけここに置く。
- **PowerShell の `Set-Content` / `Out-File` は `-Encoding utf8` を明示しないと日本語 UTF-8 を壊す**(ANSI 化)。
  日本語を含むファイルは Claude の Write/Edit か `node fs.writeFileSync(..., 'utf8')` で書く。
- PowerShell 5.1 は native コマンドの stderr を `2>&1` で error record 化する。情報メッセージをエラー扱いして
  「失敗」と誤判定し得る。**確実性重視ならビルドは Bash ツールでコマンドを直接実行**する。
- `_work/` はローカルの使い捨て作業領域(gitignored)。**コミットしない**。

## 8. ドキュメント

ドキュメントの所在・命名・形式を全プロジェクト共通でそろえる。

### 置き場所
- 複数プロジェクトに跨る**横断**ドキュメント → リポジトリ直下の `docs/`。
- そのプロジェクトの**正本**ドキュメント → そのプロジェクト直下の `docs/`。
- 入口は `docs/index.html`(または各バケットの `index.html`)。

### トップ階層 = 3 バケット
ドキュメントは作業の性質で 3 つに分ける(ブランチのアンブレラ接頭辞と同じ粒度・同じ判断で選ぶ):
- `docs/issue/` … 課題対応(不具合・改善要望など問題駆動)。
- `docs/future/` … 将来構想・検討(まだ確定していない探索)。
- `docs/project/` … プロジェクト/取り組み単位のまとまった作業。

### バケット配下の構造(5 桁連番 + フェーズ)
- 各まとまりは `docs/<bucket>/<5桁ゼロ埋め連番>/` に 1 ディレクトリ(例 `docs/issue/00001/`)。
  **連番はバケットごと**に採番し、**5 桁ゼロ埋めは必須**(後から順序が狂わないため)。
- **採番手順(重複防止)**: 新しいまとまりを作るときは、**そのバケットの既存ディレクトリと台帳 `index.html` を必ず確認し、
  存在する最大番号 + 1** を採る(空なら `00001`)。**既存番号の再利用・重複は禁止**。同時に複数を起票するときは連続して採番し、
  採番したら台帳 `index.html` にもその場でエントリを追加して次の採番がズレないようにする。
  (この採番は `scripts/new-doc.mjs` で機械化できる。下記「scaffold」参照。)
- 配下は**フェーズ番号付き**(必要な段階だけ使う):
  `01-investigation`(調査) → `02-improvement`(改善) → `03-design-proposals`(設計案) →
  `04-validation-audit`(検証・洗い出し) → `05-final-design`(最終設計) → `06-implementation-record`(実装記録)。
- ナビは 3 階層: `docs/<bucket>/index.html`(台帳=一覧) → `<bucket>/<5桁>/index.html`(その案件の目次) → 各フェーズ。
- まとまりの番号は、対応するブランチ(`<bucket>/<slug>-...`)とリリース版に連動させる。

### 常設リファレンス(reference/ 区画 — 作業エピソードと分離)
作業エピソード(バケット)とは別に、**持続的に保つ文書**(仕様・要件・契約・設計・使い方・背景)は
各 docs の `reference/` 区画に置き、**その場で更新(in-place)**する(連番バケットには入れない)。
- 下位区分は**上流 → 下流の順に `<2桁ゼロ埋め>_<名前>`**(番号 + アンダースコアで始める。順序が読み取れることを優先):
  - `00_requirements` … 要求・要求分析(何のためか)
  - `01_spec` … 仕様・契約
  - `02_architecture` … アーキテクチャ・設計の構造(arc42 + C4 等)
  - `03_decisions` … 設計決定の記録(ADR)
  - `04_usage` … 使い方(`usage/` 内でも 手順=how-to と 参照値=reference は分ける)
  - `05_explanation` … 背景・なぜ(補足理解)
  - ※ この番号付き区分は**例外的にアンダースコア**を使う(他の命名はハイフン)。
- **設計の「決定」は ADR**(Architecture Decision Records)で `03_decisions/` に置く。**追記専用・編集せず supersede**
  (後継 ADR で置き換え、過去の決定と理由を残す)。テンプレは Title / Status / Context / Decision / Consequences。
- 横断の常設は repo 直下 `docs/reference/`、プロジェクト固有は `<project>/docs/reference/`。
- **基本は上記フル構造**。区分が過剰になりそうな小規模なケースでも**勝手に省略せず、どうするかユーザーに判断を仰ぐ**(既定で緩和は入れない)。

### 命名(コミット/ブランチと同じ文字規則)
- まとまり: `docs/<bucket>/<5桁ゼロ埋め>/<2桁>-<phase>.html`。
- 横断 `docs/` 直下のトピック文書: `<2桁>-<kebab>.{html|md}`(十の位でカテゴリ分け)。
- 小文字英数字 + ハイフンのみ。連番は 5 桁ゼロ埋め必須。

### 形式
- 図解・設計案・調査・実装記録など読ませるもの = **HTML**(自己完結・インライン CSS・インライン SVG)。
- 軽量な要件・進捗・短メモ = **Markdown 可**(図やレイアウトが要るものは HTML)。
- **外部 CSS / CDN / 共有 CSS ファイルに依存しない**。各ドキュメントは単体で開けること(隔離環境でも成立させる)。

### 分割 → 統合ワークフロー(AI 編集前提)
長文・複数章のドキュメントは、AI が部分編集しやすい粒度に**分割**し、最後に 1 HTML へ**統合**する。
読者にはあくまで統合済みの 1 HTML だけを見せる(自己完結 HTML の原則は維持)。

- **分割編集**: 章節ごとに `_parts/NN-section.html` として `<section>` 単位で保存する。
  ファイル名は 2 桁ゼロ埋め連番 + ケバブの章名(例 `01-background.html`、`02-design.html`)。
  各 part は `<section>` 1 つ分の本文だけを持ち、`<html>`/`<head>`/`<body>` は含めない。
- **小規模は分割しない**: 1〜2 章で済む短いドキュメントは分割せず、本文を 1 ファイル内に書く。
  章節を `<details>/<summary>` で折りたためる構造にしておくと AI が部分編集しやすい。
- **中間ファイルはコミットして参照しない**: `_parts/` 配下はコミットして履歴に残すが、
  ナビゲーション(`index.html` や台帳カード、フェーズリンク)からは**参照しない**。
  読者の入口は統合済みの 1 HTML だけ。
- **統合スクリプト同梱**: 各リポに `scripts/assemble-doc.mjs` を置く(project-template に雛形あり)。
  `node scripts/assemble-doc.mjs <_parts ディレクトリ> -o <出力 HTML>` で順序結合する。
  Node のみで外部依存なし。雛形 `docs/_templates/_parts/{00-head,01-section,99-foot}.html` から始める。
- スクリプトの使い方詳細・分割粒度の指針は ai-skills の `assembling-html-doc` skill を参照。

### メタ・見出し・図
- 冒頭に `<title>`(識別名)と `<div class="meta">`(対象 / ブランチ / 関連 / ステータス。ステータスはクラスで表現)。
- 見出しは h1(1 つ)> h2 > h3。図は `<figure>` + **インライン SVG** + キャプション。
- 更新履歴はドキュメントに持たせず Git に委ねる。

### 表示レビュー必須フロー(全 HTML 必須)
HTML を作成・更新したら、SVG の有無に関わらず**視覚レビュー**を必ず行う。
内容レビュー(後述の 3 体レビュー)とは責務を分け、**視覚レビューを先**にやる。

1. **各 part 単体**(分割編集の場合): `_parts/NN-*.html` を編集したら、当該 part を
   雛形ヘッダだけで包んだプレビューを開き、目視で表示崩れ・図のはみ出し・文字切れがないか確認する。
2. **統合後の 3 モード**: 完成 HTML(分割しない場合はそのファイル)を、
   **ライト / ダーク / 印刷プレビュー** の 3 モードでスクリーンショットを撮り、目視で確認する。
   - ライト: 既定のまま開く
   - ダーク: 右上トグルでダークに切替えて表示
   - 印刷: ブラウザの印刷プレビュー(`⌘P` / `Ctrl+P`)を開く。ダーク状態でも `@media print` でライトに戻ること、トグルが非表示になることを確認
3. **崩れていたら直して再撮影**。`viewBox`・サイズ・折り返し・座標・コントラストを修正し、直るまで「完了」と言わない。
4. SVG は環境差で特に崩れやすい。図の線・テキストが両テーマ・印刷の 3 モードすべてで読めることを毎回確認する。

**目視チェックの観点(必須)**: スクリーンショットを「撮る」だけでなく、次の観点で必ず**見る**こと。
- **乱れがない**: 図・表・文字に表示崩れ(はみ出し・重なり・切れ・折り返し不良)がない。
- **文字と図が分かりやすい**: 図中の文字が背景と十分なコントラストで読め、線・矢印・ラベルの対応が一目で追える。
- **大きさが整っている**: 文字サイズ・図の縮尺・余白がページ内で揃っており、極端に小さい/大きい要素がない。

視覚レビューは**表示崩れ・図の可読性専用**、後述の 3 体レビューは**内容・規約準拠・可読性の総合**で、
両者の責務を分離する。順序は **視覚レビュー → 3 体レビュー → ユーザー提示**(後述レビュー節と一致)。

### レビュー(新規ドキュメントはユーザー提示前・必須)
- **新規に作成する(または内容を実質的に書き換える)ドキュメント**は、ユーザーに提示する前に、
  まず **視覚レビュー(上節)** を済ませ、その後 **厳しめのレビューエージェントを最低 3 体**立てて、
  それぞれ異なる観点(§8 準拠 / 内容の正確さ・リンク整合 / 可読性 等)で指摘を出させる。
  指摘を**修正してからコミットし、その上でユーザーに提示**する。
- 順序: **視覚レビュー → 3 体レビュー → ユーザー提示**。表示崩れは 3 体レビューより前に解消する。
- コミットは**修正前 → 修正後が追える細かい粒度**で行う(レビュー反映を 1 コミットにまとめない)。
- **コスト最適化(必須設定)**: 3 体のレビューエージェントは既定で**高速・低コストモデル**で走らせる。
  限定された文書への観点別の批評には十分で、フロンティアモデルは不要。文書の作成・修正反映は通常のセッションモデルで行う。
- モデルは切替可能。重要度が高い文書ではレビューを上位モデルへ上げてよい。
- 指摘は**客観的事実に徹する**——題材の是非(合法/違法・好ましい/好ましくない)など法的・道徳的な評価や「リスク」断定をしない(§10 参照)。

### 共通テーマ(ライト既定・ダーク切替・印刷対応)
共有 CSS ファイルは作らず、各 HTML が同じ CSS 変数を**インライン**で定義する。
**既定はライト**。`:root` にライト値を置き、`html[data-theme="dark"]` でダーク値に上書きする。
ページ右上にトグル(☀ ライト / ☾ ダーク)を配置し、選択は `localStorage` に保存して次回も維持する。
`@media print` でダーク指定を**強制的にライトへ戻し**、トグルは非表示にする(PDF 化・紙印刷ですぐ読める状態)。

ライト(既定):
```
--bg:#ffffff; --panel:#f6f8fa; --line:#d8dee4; --tx:#1f2328; --tx2:#59636e;
--acc:#0969da; --ok:#1a7f37; --warn:#9a6700; --err:#d1242f;
--code-bg:#f6f8fa; --code-tx:#0550ae; --th-bg:#f6f8fa;
```

## 9. 課題・バグ報告(Issue)

Issue を立てるとき(**特に別リポジトリに対して立てるとき**)は、**再現できる動作環境とバージョン情報を必ず添える**。
「自分の環境だけで起きる/直る」状態を避け、受け手が再現・切り分けできる報告にする。エージェントが Issue を起票・代筆する場合も同様。

**必ず含める情報**(カテゴリ。具体値は Issue 本文に書き、このファイルには書かない):
- **OS / プラットフォーム**(種別とバージョン)。
- **シェルと実行形態**(対話 / 非対話、TTY の有無、CI かどうか)。
- **関連ツールのバージョン**(その挙動を示すツール / CLI / ランタイム / 言語の正確な版)。判明していれば認証方式やモードも。
- **最小の再現手順**(実際に叩いたコマンド)と**実際の出力**(エラー文はそのまま引用)。
- **期待した挙動 / 実際の挙動**、判明していれば**回避策**。
- どのリポジトリ・どの作業中に遭遇したか(文脈)。

環境情報を取得するコマンド例(対象に合わせて選ぶ): `uname -a` / `sw_vers` / `echo $SHELL` / `<tool> --version` など。
- **本文には恒久的に正しい「報告に含めるべきカテゴリ」だけを書く**。特定のツール名・版番号・パスなどの具体値はここに書かない(各 Issue 本文に書く)。

## 10. AI 出力・レビューの中立性(客観的事実のみ)

AI が生成し、また AI に読ませる文脈(レビュー指摘・監査/調査レポート・棚卸し・自動生成ドキュメント等)は、
**客観的事実のみ**を述べる。法的・道徳的な評価や「リスク」の断定をしない。

- レビュー・監査・調査では、**事実**(技術・構成・更新状況・依存とその版・秘密情報の混入有無・再現する不具合 等)を示すに留め、
  題材の是非(合法/違法、好ましい/好ましくない)を**判断しない**。是非は法域・意図・文脈に依存し、ツールやレビューの能力外。
- 公開されている情報や、学習・研究目的の題材に対して、法的/道徳的なラベル(「法的リスク」等)を**付けない**。
- **理由**: そうしたラベルは AI に渡る文脈を汚染し、後続の AI を過剰な拒否へ priming して、正当な作業・対話・協働を壊す。
  生成物が自分自身の制約になる失敗モードを避け、事実に徹することで文脈の信頼性と将来の協働可能性を保つ。
- **例外**: 他者への明白な加害が問題になる場合に限り、評価ではなく**事実として**述べる。それ以外で是非を断定しない。
- 本原則は §8「レビュー」にも適用する(レビューエージェントの指摘も客観的事実に徹する)。

## 11. 教訓の書き戻し(CLAUDE.md を育てる)

同じ失敗・同じ注意をセッションを跨いで繰り返さないため、各リポジトリの CLAUDE.md に**「教訓(再発防止)」節**を設け、
AI が育てる。CLAUDE.md は起動時に必ず読まれるため、ここに書いた教訓は次のセッションから自動的に効く。

- **発動条件**: 同じ内容でユーザーから **2 回目の注意**を受けたとき、または同じ原因のビルド / 実行エラーに
  **2 回目に遭遇**したとき、教訓節への追記案を提示する(**1 回目では書かない** — ノイズ防止)。
- **形式**: 1 教訓 = 1 エントリ「`[YYYY-MM-DD] 事象 → 原因 → 次からどうする`」(1〜3 行)。
- **追記はユーザーに一言確認してから**行う。AI が書き込んでよいのは**この節だけ**
  (CLAUDE.md の他の節は「地図」として保ち、勝手に膨らませない)。
- **棚卸し**: 解決済み・陳腐化した教訓は削除を提案する。複数プロジェクトに当てはまると分かった教訓は、
  本ファイル(AGENTS-SHARED)への**昇格を提案**する(冒頭の「別プロジェクトにそのまま貼れるか?」の判定に従う)。
